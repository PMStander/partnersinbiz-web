Three approaches to the account picker

Pick the one that feels right — trade-offs are below each option

A

Pending + Modal on accounts page Recommended

OAuth callback saves all available options (pages + personal) to a temporary social_oauth_pending Firestore doc, then redirects to accounts page with ?picker=nonce. The accounts page detects this and shows a modal. User picks multiple, one is set as default. Each selection becomes its own social_accounts doc. One card per platform on accounts page, separate entries on compose.

Pros

  • Fits existing architecture perfectly
  • No new pages to route
  • Each sub-account has its own accountId (compose page just works)
  • Account-resolver already handles multiple docs per platform

Cons

  • Pending doc needs cleanup if user abandons
B

Separate picker page

OAuth callback redirects to /admin/social/accounts/pick?token=nonce — a dedicated page showing the options. User picks, submits, then lands on accounts page. Cleaner separation of concerns but adds a new route.

Pros

  • Clean URL-driven flow
  • Easy to deep-link or revisit

Cons

  • Extra page to build and route
  • Back button can cause confusion
  • More code for essentially the same UI
C

Sub-accounts array inside one doc

Store all sub-accounts inside a single social_accounts doc as a subAccounts[] array. Compose page references accountId + subAccountIndex. One doc per platform always.

Pros

  • Fewest Firestore docs
  • Natural grouping by platform

Cons

  • Breaks existing account-resolver (built around one doc = one account)
  • Compose page references become more complex
  • Scheduled posts need updating when sub-accounts change
